# 项目详情

本文档面向开发者，描述当前仓库在 `0.3.0` 版本下的真实状态。它不重复用户手册中的操作说明，重点解释系统分层、运行链路、关键模块、配置注入、已落地能力和维护约束。

## 1. 先建立正确心智模型

当前仓库已经不是“一个 Python 脚本配一个 GUI”的结构，而是一条明确的多层主链：

- React 前端负责桌面 UI、输入编排和任务展示。
- Python 宿主负责 Qt 窗口、本地 HTTP API、配置管理、文件上传和桥接调用。
- TypeScript Runtime 是 Agent 主脑，负责会话、模式判定、规划、执行编排、运行预算、状态存储和任务审计。
- Python 工具层负责 observation 首选路径、DOCX 写操作和最终物化。
- TS 保留 native observation fallback，但对上层统一暴露共享 schema。

一句话概括：

- 决策在 `src/ts/`
- 宿主在 `src/python/`
- 交互在 `src/frontend/`
- 真实 DOCX 改写落在 Python 工具链

## 2. 当前产品能力边界

当前已稳定存在的能力包括：

- 聊天式问答：无文档时也可以做普通对话。
- 文档观察：对已绑定 DOCX 进行结构分析和问答。
- 文档执行：对 DOCX 执行格式调整和结构修改。
- 歧义澄清：无法确定范围时先追问，不直接猜测执行。
- 模板任务：在模板工作区中导入 DOCX、选择模板 JSON、执行模板链路。
- 本地 API：支持会话、消息、模板、配置和输出目录相关操作。

当前不应假设存在的能力：

- 多人协作服务端
- 远程云端托管
- 通用 Office 格式支持（当前主目标仍是 `.docx`）
- 前端直接做 Agent 决策

## 4. 启动链路

推荐入口是 `scripts/launch_gui.py`。

启动流程：

1. `launch_gui.py` 检查 GUI 依赖、`node`、配置文件和构建产物。
2. 校验 `src/ts/dist/runtime/cli.js` 与 `src/frontend/dist/index.html` 存在。
3. 检查 `src/ts/src` 是否比 `src/ts/dist` 更新，避免前端或运行时使用旧构建产物。
4. 检查通过后，调用 `scripts/run_gui.py`。
5. `run_gui.py` 进入 `src.python.gui.web_window.run()`。
6. `web_window.py` 创建 `WebApiServer`，启动本地 HTTP 服务，再由 `QWebEngineView` 加载前端页面。

这里有两个容易踩坑的点：

- 桌面 GUI 读取的是构建产物，不是源码目录。
- Qt WebEngine 运行时数据不写系统默认目录，而是写项目内 `.tmp/qtwebengine/`。

## 5. 前端实际行为

### 5.1 路由

前端当前只有两个主要界面：

- `/`：聊天工作区
- `/templates`：模板工作区

这不是 React Router，而是 `App.tsx` 里基于 `window.history` 的轻量切换。

### 5.2 草稿会话与待发送文件

`useChatStore.ts` 当前实现了一个很关键的交互模型：

- 新建对话先生成前端草稿会话 `draft:*`
- 选择 `.docx` 时，只保存为待发送文件
- 点击发送时，才会按顺序执行：
  - 创建远端会话
  - 上传文档
  - 提交消息
- 首轮成功后，草稿会话会被真实会话替换

这意味着：

- “选中文件”不等于“已经绑定到后端会话”
- 后端 `/messages` 仍然要求非空 `content`
- “仅文件发送”是由前端补默认引导语实现的，不是后端放宽了契约

默认补的提示语是：

- `请先分析我上传的文档并开始处理。`

### 5.3 异步 job 展示

前端不会把一次 HTTP 请求的完成直接等价为任务完成。它会：

- 先创建本地临时 job
- 提交 `/messages/async`
- 轮询 `/message-jobs/{jobId}`
- 根据返回的远端步骤状态更新 UI

这也是为什么同步接口超时时，GUI 仍然可能在后续看到任务完成结果。

## 7. Python ↔ TS bridge 机制

### 7.1 TS Agent Bridge

`src/python/api/ts_agent_bridge.py` 的工作方式是：

1. 为每次请求在 `.tmp/` 下创建独立临时目录。
2. 把请求写入 `input.json`。
3. 调用 `node dist/runtime/cli.js --input-json ... --output-json ...`。
4. 读取 `output.json`。
5. 清理临时目录。

这个桥接层负责：

- 会话创建
- 会话列表读取
- 提交用户回合
- 绑定文档
- 查询回合运行状态
- 更新标题
- 删除会话

### 7.2 模板 Bridge

`src/python/api/template_bridge.py` 的模式与 TS bridge 类似，但目标 CLI 不同：

- CLI 路径是 `dist/templates/template-cli.js`
- 输入是模板任务请求
- 输出是模板执行报告
- 错误中会尽量保留 `stage`、`stage_timings_ms` 和 `stderr_summary`

### 7.3 环境变量注入

Python 通过 `build_ts_agent_env()` 把模型配置注入给 TS 进程，例如：

- `TS_AGENT_CHAT_BASE_URL`
- `TS_AGENT_CHAT_API_KEY`
- `TS_AGENT_CHAT_MODEL`
- `TS_AGENT_PLANNER_BASE_URL`
- `TS_AGENT_PLANNER_API_KEY`
- `TS_AGENT_PLANNER_MODEL`
- `TS_AGENT_STEP_TIMEOUT_MS`
- `TS_AGENT_TASK_TIMEOUT_MS`
- `TS_AGENT_PYTHON_TOOL_TIMEOUT_MS`
- `TS_AGENT_MAX_TURNS`
- `TS_AGENT_SYNC_REQUEST_TIMEOUT_MS`

维护预算模型时，必须同时检查：

- `config.example.json`
- `src/python/core/model_config.py`
- `src/python/api/ts_agent_bridge.py`
- `src/python/api/template_bridge.py`

## 8. TS 会话与回合主链

真正的 Agent 主链入口在 `src/ts/src/runtime/session-service.ts`。

单轮处理流程大致是：

1. `submitUserTurn()` 创建 turn run 记录。
2. 把用户消息写入会话状态库。
3. 调用 `model-gateway` 决定模式。
4. 按模式进入 `chat`、`inspect`、`execute` 或澄清分支。
5. 在需要时读取 observation、构造 `DocumentIR`、调用 `engine.run()`。
6. 追加 assistant 消息。
7. 更新 turn run 状态与步骤。

当前 turn run 会跟踪：

- `status`
- `summary`
- `mode`
- `goal`
- `steps`
- `error`

因此前端与 Web API 能够展示“判定模式”“读取文档”“规划并执行步骤”等状态。

## 9. 模式判定与行为约束

当前单轮模式分为：

- `chat`：纯对话回复
- `inspect`：读取文档并生成观察型回复
- `execute`：执行写操作
- `needsClarification=true`：先向用户澄清

需要坚持的约束：

- 当模式不是 `chat` 时，必须有已绑定文档。
- 歧义需求优先澄清，不要偷偷收窄范围。
- 不要把规划逻辑重新塞回 Python 或前端。
- TS 是唯一决策主脑，Python 负责执行。

当前代码还内置了一个重要规则：

- 如果用户说“正文”，且没有明确排除列表项，执行目标会默认同时覆盖普通正文和 `list_item`，并在结果里附带说明。

## 10. Runtime 与执行模型

`src/ts/src/runtime/engine.ts` 提供两种运行模式：

- `plan_once`
- `react_loop`

`createMvpRuntime()` 会根据配置组装：

- Planner 或 ReAct Planner
- Tool Registry
- Validator
- Audit Store
- Document Tooling Facade
- Document Execution Facade

### 10.1 `plan_once`

流程：

1. 先让 Planner 一次性产出完整计划。
2. 扩展 selector。
3. 逐步执行。
4. 校验结果。
5. 物化最终文档。

### 10.2 `react_loop`

流程：

1. ReAct Planner 每轮只决定下一步。
2. 执行单步。
3. 将 observation 写入 trace。
4. 继续下一轮，直到 `finish` 或失败。

当前示例配置默认偏向 `react_loop`。

## 11. Observation 统一收口

当前 observation 已经统一到 facade 边界，不应再让业务层直接感知两套类型。

关键模块：

- `document-tooling/facade.ts`
- `document-tooling/observation.ts`
- `document-tooling/observation-policy.ts`
- `tools/docx-observation-schema.ts`
- `tools/python-tool-client.ts`
- `tools/docx-observation-tool.ts`

真实策略是：

1. 优先走 Python observation。
2. 若命中允许降级的依赖缺失或特定解析失败，再退到 TS native observation。
3. 无论哪条路径，都向上输出同一份 observation schema。

这样做的目的：

- session、runtime、template 只消费一种 observation 结构
- fallback 被限制在策略点，而不是散落在上层业务代码里
- 上层无需通过类型断言区分 Python / native

## 12. 文档语义与选择器

当前 observation 不只是扁平文本，它还包含结构语义，供选择器扩展、模板分类和澄清使用。

常见字段包括：

- `document_meta.total_paragraphs`
- `document_meta.total_tables`
- `paragraphs[].id`
- `paragraphs[].text`
- `paragraphs[].role`
- `paragraphs[].heading_level`
- `paragraphs[].list_level`
- `paragraphs[].run_ids`
- `nodes[]`

当前稳定的语义范围包括：

- `body`

## 15. 状态存储与审计

当前项目有两套 SQLite 相关状态：

- 会话状态：由 `SqliteAgentStateStore` 负责
- 任务审计：由 `SqliteTaskAuditStore` 负责

它们分别服务于不同目标：

- 会话状态用于恢复会话、消息和回合展示
- 审计数据用于记录执行过程、事件和 ReAct trace

不要把两者混为一谈，也不要在改一个时忽略另一个的兼容性。

## 16. 同步、异步与超时语义

这是当前系统最容易被误解的部分。

### 16.1 配置字段

当前预算模型主要由以下字段驱动：

| 字段 | 默认值 | 语义 |
| --- | --- | --- |
| `planner.timeout_ms` | `null` | 单次模型请求超时 |
| `planner.step_timeout_ms` | `60000` | 默认单步执行预算 |
| `planner.task_timeout_ms` | `null` | 整轮任务总预算 |
| `planner.python_tool_timeout_ms` | `null` | 单次 Python 工具调用预算 |
| `planner.max_turns` | `24` | `react_loop` 最大轮数 |
| `planner.sync_request_timeout_ms` | `300000` | 同步 HTTP 等待上限 |
| `planner.runtime_mode` | `react_loop` | 默认运行模式 |

### 16.2 语义区别

- `timeout_ms` 不等于整个任务超时
- `step_timeout_ms` 是 Runtime 内部执行常用预算
- `task_timeout_ms` 控制整轮任务寿命
- `python_tool_timeout_ms` 只约束 Python 工具调用
- `sync_request_timeout_ms` 只约束宿主这一跳愿意等多久

### 16.3 同步接口不是任务真相

`POST /messages` 的行为是：

- 创建后台 job
- 在同步窗口内等待
- 如果到时没完成，返回 `E_SYNC_REQUEST_TIMEOUT`

它不意味着：

- 后台任务一定失败
- 运行时已经终止

真正的任务真相要以后续 job 状态为准：

- `running`
- `waiting_user`
- `failed`
- `completed`

## 20. 维护建议

- 文档与代码冲突时，以代码和测试为准，再回头修文档。
- 改输入交互时，先确认是否真的需要修改后端协议，优先在前端 store 收口。
- 改会话链路时，同时检查前端 store、`web_api.py`、bridge 和 `session-service.ts`。
- 改 observation 时，只在 facade 与 policy 边界处理 fallback，不要把分叉散到上层。
- 改写操作时，同时检查 planner、validator、style operation、Python 工具实现和测试。
- 改超时或预算时，把配置、Python 注入层、TS runtime 和文档一起更新。
- 改模板链路时，注意模板工作区、模板 API、template bridge、TS template 子系统和回归测试要保持一致。
