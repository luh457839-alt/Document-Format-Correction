# 项目详情

本文档面向开发者，用来快速建立对当前仓库的正确心智模型。它不重复用户手册中的操作说明，重点讲清楚系统边界、核心模块、关键链路和维护要点。

## 项目定位

当前仓库已经收敛为一条明确主链：

- TypeScript 是 Agent 主脑，负责会话状态、模式判定、规划、执行编排和审计。
- Python 是宿主层和工具执行层，负责桌面 GUI、本地 Web API、TS CLI 桥接、DOCX 观察与写回。
- React 前端只负责桌面 UI 和输入编排，不负责核心 Agent 决策。

换句话说：

- “理解用户要做什么”主要发生在 `src/ts/`
- “真正把 DOCX 改出来”主要发生在 Python 工具层
- “把用户输入送进主链”发生在 `src/frontend/` 和 `src/python/gui/`

## 当前架构

### 1. 前端层

目录：`src/frontend/`

职责：

- 提供桌面内嵌 Web UI
- 管理会话列表、消息区、输入区和设置面板
- 通过 `/api/*` 调用 Python 宿主
- 编排“文本 + 文件”的统一发送体验

当前输入行为有一个需要特别记住的约束：

- 选择 `.docx` 文件时，前端只会把文件保存在待发送状态
- 点击发送时，前端才会统一触发创建会话、上传文档和提交消息
- 支持三种输入：仅文本、文本 + 文件、仅文件
- 仅文件发送时，前端会自动补一条固定引导语，再提交首轮请求

关键文件：

- `src/frontend/store/useChatStore.ts`
- `src/frontend/components/chat/InputArea.tsx`
- `src/frontend/components/chat/ChatSpace.tsx`
- `src/frontend/services/api.ts`

### 2. Python 宿主层

目录：`src/python/gui/`、`src/python/api/`

职责：

- 启动桌面窗口
- 托管本地 HTTP API
- 把前端请求桥接到 TS CLI 和 Python 工具
- 管理本地配置、上传文件和静态资源

关键文件：

- `src/python/gui/web_window.py`：桌面 GUI 入口
- `src/python/gui/web_api.py`：本地 HTTP API
- `src/python/api/ts_agent_bridge.py`：Python ↔ TS CLI 桥接
- `src/python/api/python_tool_runner.py`：Python 工具统一执行入口

### 3. TypeScript Agent 层

目录：`src/ts/`

职责：

- 维护会话和运行态
- 判定当前回合属于 `chat`、`inspect`、`execute` 或澄清场景
- 调用模型并生成下一步计划
- 执行工具、处理回滚、记录审计

关键文件：

## 关键链路

### 启动链路

1. `scripts/launch_gui.py` 做启动前检查。
2. 检查 Python GUI 依赖、Node.js、`config.json`、TS 构建产物和前端构建产物。
3. 检查通过后，调用 `scripts/run_gui.py`。
4. `run_gui.py` 进入 `src.python.gui.web_window.run()`。
5. Python 宿主启动本地 Web API，并在桌面窗口中加载 `src/frontend/dist`。

### 会话与消息链路

1. 前端或调用方调用 `/api/sessions`、`/api/sessions/{id}`、`/api/sessions/{id}/messages`，长任务场景也可以走 `/api/sessions/{id}/messages/async`。
2. `web_api.py` 把请求转发给 `ts_agent_bridge.py`。
3. `ts_agent_bridge.py` 调用 TS CLI。
4. `session-service.ts` 读取和更新会话状态。
5. `model-gateway.ts` 决定本轮模式。
6. `engine.ts` 在需要时执行规划和工具调用。

### 超时与运行预算模型

当前超时治理不是单一常量，而是 7 个互相配合的预算字段：

| 字段 | 当前默认值 | 语义 |
| --- | --- | --- |
| `planner.timeout_ms` | `null` | 单次 LLM 请求超时；主要作用在 Planner / ReAct 的模型调用层。 |
| `planner.step_timeout_ms` | `60000` | 默认执行步超时；未单独覆盖时，是运行时最常见的单步预算。 |
| `planner.task_timeout_ms` | `null` | 整轮任务总预算；`null` 表示不额外施加整轮硬上限。 |
| `planner.python_tool_timeout_ms` | `null` | 单次 Python 工具调用超时；`null` 时继承 `step_timeout_ms` 的语义。 |
| `planner.max_turns` | `24` | `react_loop` 模式下的最大 ReAct 轮数。 |
| `planner.sync_request_timeout_ms` | `300000` | 同步入口的外层等待上限，用来限制“HTTP 这一跳愿意等多久”。 |
| `planner.runtime_mode` | `react_loop` | 当前示例配置与主推荐链路使用的运行模式。 |

这几层语义要分开理解：

- `planner.timeout_ms` 限制的是单次模型请求，不等于整个任务超时。
- `planner.step_timeout_ms` 限制的是默认执行步预算，偏向 runtime 内部的单步执行。
- `planner.task_timeout_ms` 只负责整轮任务预算，不直接决定某一次 HTTP 请求是否返回。
- `planner.python_tool_timeout_ms` 只约束 Python 工具调用；未设置时，沿用单步预算。
- `planner.max_turns` 控制 ReAct 循环的最大轮数，不是 wall-clock 时间预算。
- `planner.sync_request_timeout_ms` 控制同步等待时长，超时时应理解为“外层不再继续等”，而不是“任务已经失败”。

### 同步与异步入口差异

当前入口语义已经明确分层：

- `/messages` 是兼容同步入口。它会创建后台 job，然后在 `sync_request_timeout_ms` 内等待结果。
- `/messages/async` 会立即返回已受理的 job，调用方后续通过 `message-jobs/{jobId}` 轮询状态。
- 桌面 GUI 的长任务体验本质上也是基于后台 job 状态，而不是把“一次 HTTP 成功 / 失败”当成全部真相。

因此，任务状态应优先理解为运行态：

- `running`：任务仍在执行。
- `waiting_user`：任务暂停，等待用户确认或补充信息。
- `failed`：任务明确失败。
- `completed`：任务已经完成。

`/messages` 返回 `504` 且错误码为 `E_SYNC_REQUEST_TIMEOUT` 时，语义是“同步等待超时，但后台任务可能仍在继续”。只有后续任务状态进入 `failed`，才能判定为真正执行失败。

### Python 宿主层里的 bridge 语义

Python 宿主层当前同时承接同步等待和后台运行，两者不要混淆：

- 同步 bridge 调用默认应用 `sync_request_timeout_ms`，来源是 `src/python/api/ts_agent_bridge.py` 中的默认 bridge 选项。
- `web_api.py` 的同步 `/messages` 路径也是先创建后台 job，再按 `sync_request_timeout_ms` 等待 job 状态。
- 后台 job / 异步调用会以 `timeout_sec=None` 调用 `submit_agent_turn(...)`，允许 TS runtime 继续跑完。
- 后台期间的状态刷新依赖 `get_turn_run_status(...)`，因此外层等待超时不应被写成执行失败。

### 文档导入与发送链路

当前这条链路和旧版本相比，最大的变化是“文件先选中，发送时统一提交”。

流程如下：

1. 用户在前端选择 `.docx`
2. `useChatStore.importDocument()` 只保存待发送文件
3. 用户点击发送
4. 若当前是草稿会话，先调用 `createSession()`
5. 调用 `attachDocument()`
6. 再调用 `submitTurn()`
7. 首轮提交成功后，草稿会话切换成远端会话

需要特别注意两点：

- `/api/sessions/{id}/messages` 仍保持 `{ content: string }` 的请求结构，没有为了“仅文件发送”改后端契约
- 仅文件发送是通过前端补固定引导语实现的，不是放宽后端空文本限制

### 文档观察链路

1. 会话已绑定 DOCX
2. TS 通过 Python tool runner 调用观察工具
3. Python 解析文档并返回结构化 observation
4. TS 结合 observation、会话上下文和当前目标生成答复

### 文档执行链路

1. TS 先构建 `DocumentIR`
2. Planner / ReAct 生成步骤
3. 若步骤使用 `targetSelector`，先扩展成具体 `targetNodeId`
4. TS 通过 Python tool runner 调用写工具
5. Python 在 scratch 文档上执行真实修改
6. 全部成功后，统一物化到最终 DOCX 并写入 `output/`

## 当前行为约束

### 歧义需求先澄清

像「正文」「这部分」「下面这块」这类天然有范围歧义的表述，不应直接猜测执行。当前主链的预期行为是：

- 先 inspect 文档
- 再返回 `【需求澄清】`
- 让用户补全范围后再进入执行

### TS 是唯一决策主脑

不要把“规划逻辑”重新塞回 Python 宿主或前端。当前边界是：

- TS 决定做什么
- Python 执行工具
- 前端负责交互和编排

### 前端不改后端契约

最近一轮输入区改造刻意保持了后端接口不变。开发时应继续默认：

- `/messages` 仍要求非空 `content`
- `/attach-document` 不额外携带聊天文本
- 组合输入由前端 store 编排，而不是靠 Web API 新增临时协议

## 文档结构语义

当前 observation 不只返回扁平文本运行，还返回段落级结构信息。这是后续范围定位和歧义澄清的基础。

常见字段包括：

- `paragraphs[].id`
- `paragraphs[].text`
- `paragraphs[].role`
- `paragraphs[].heading_level`
- `paragraphs[].list_level`
- `paragraphs[].run_ids`

当前内置选择器范围包括：

- `body`
- `heading`
- `list_item`
- `all_text`
- `paragraph_ids`

需要记住：

- `body` 不包含项目符号 / 编号段落
- 项目符号 / 编号段落会单独视为 `list_item`
- 当自然语言目标无法明确落到某个范围时，应优先澄清，而不是偷偷收窄

## Python 边界与错误分类

Python 边界当前已经显式化，维护时不要再把这些错误重新折叠成一类“执行失败”：

- `E_PYTHON_TOOL_START_FAILED`
- `E_PYTHON_TOOL_EXIT_NONZERO`
- `E_PYTHON_TOOL_OUTPUT_MISSING`
- `E_PYTHON_TOOL_OUTPUT_INVALID_JSON`
- `E_PYTHON_TOOL_OUTPUT_INVALID_SHAPE`
- `E_PYTHON_DEPENDENCY_MISSING`

当前依赖降级也有明确约束：

- 只有 Python 侧明确返回 `E_PYTHON_DEPENDENCY_MISSING`，才允许某些 observation 场景退到 TS 原生 parser
- 不再依赖模糊的错误消息做正则猜测

## 配置与模型

## 维护建议

- 改输入交互时，先确认是否真的需要改 Web API 契约；能在前端 store 编排解决的，优先不要扩接口。
- 改会话或文档链路时，优先检查 `src/frontend/store/useChatStore.ts`、`src/python/gui/web_api.py`、`src/python/api/ts_agent_bridge.py`、`src/ts/src/runtime/session-service.ts` 这四层是否一致。
- 改 DOCX 写操作时，同时考虑 observation、IR、tool runner、写回脚本和测试覆盖，不要只改单点。
- 改错误处理时，尽量保留根因类别，不要把细粒度错误重新包装成模糊提示。
- 改 timeout 或运行预算时，把 `config.example.json`、`src/python/core/model_config.py`、`src/python/api/ts_agent_bridge.py`、`src/ts/src/planner/llm-planner.ts`、`src/ts/src/runtime/engine.ts` 当成一组一起核对，不要只改单独一层或只改前端默认值。
- 若文档和源码冲突，以源码和测试为准，再回头更新文档。
